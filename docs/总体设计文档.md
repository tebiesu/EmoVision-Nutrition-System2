# 基于情感识别与视觉分析的个人饮食健康智能助手总体设计文档

## 1. 课题概述

### 1.1 课题背景

基于个人日常饮食健康管理与情绪状态感知需求，开发“基于情感识别与视觉分析的个人饮食健康智能助手”。系统采用前后端分离模式，Web 端用户界面基于 Vue 前端框架设计，后端管理与业务服务基于 Spring Boot 框架开发。系统围绕“采集—识别—分析—建议—追踪”构建闭环，主要功能模块包括用户健康档案管理、饮食记录与图片上传、视觉分析与食物识别、情绪识别、营养评估、个性化饮食建议、历史趋势分析与可视化展示等。

### 1.2 课题定位

本课题定位为工程实现型毕业设计，而非从零训练高精度模型的算法研究。系统重点展示需求分析、系统设计、接口接入、数据流闭环、测试验证、日志追踪与可视化展示能力。

### 1.3 总体目标

实现一个具备以下能力的 Web 系统：

1. 用户健康档案管理。
2. 饮食记录与图片上传。
3. 食物视觉识别与用户确认。
4. 情绪识别，自评为主、文本分析为辅。
5. 营养评估与风险标签生成。
6. 个性化饮食建议生成。
7. 历史趋势分析与可视化展示。
8. 审计日志与关键接口追踪。

### 1.4 设计原则

1. 系统开发为主，模型训练为辅。
2. 规则判定为核心，AI 生成作增强。
3. 可解释、可测试、可追踪、可扩展。
4. 健康辅助建议，不替代医疗诊断。

## 4. 功能需求与核心用例设计

### 4.1 功能需求概述

本系统面向关注日常饮食健康管理的个人用户，旨在通过饮食记录、食物视觉识别、情绪状态采集、营养评估与个性化建议生成，构建“采集—识别—分析—建议—追踪”的闭环健康辅助系统。系统不以医疗诊断为目标，而是为用户提供一般性、解释性、可追踪的饮食健康辅助建议。

系统核心强调以下特征：

1. 完整闭环：覆盖从用户数据采集到趋势追踪的完整流程。
2. 可解释性：建议结果需给出触发原因和依据说明。
3. 可修正性：识别结果允许用户确认或手动修正，降低模型误判影响。
4. 可测试性：核心链路和规则判定结果可通过测试验证。
5. 可扩展性：AI 能力通过接口接入，后续可替换模型服务或增加扩展模块。

### 4.2 用户角色定义

#### 4.2.1 普通用户

普通用户是系统的核心服务对象，其主要职责如下：

- 注册并登录系统
- 维护个人健康档案
- 录入每日饮食信息
- 上传食物图片并查看识别结果
- 对识别结果进行确认或修正
- 录入当前情绪状态
- 查看营养评估结果与个性化建议
- 查看热量、情绪和营养结构趋势图
- 查看每日健康总结

#### 4.2.2 管理员

管理员负责系统运行层面的监督与维护，本系统采用轻量管理员页方案，其主要职责如下：

- 查看接口调用状态
- 查看识别服务与情绪分析服务的运行情况
- 查看系统异常日志与审计日志
- 查看基础统计信息

#### 4.3.4 图片上传与视觉识别模块

- 上传食物图片
- 创建视觉识别任务
- 获取识别候选结果

### 4.4 核心业务流程

1. 用户完成注册登录后，首先录入个人健康档案。
2. 用户新增一条饮食记录，并填写餐次、时间和文字描述。
3. 用户上传当前餐食图片，系统调用视觉识别接口识别食物类别。
4. 系统返回识别候选结果及置信度。
5. 用户对识别结果进行确认、修正，或手动补录缺失食物项。
6. 用户进行情绪自评，并可补充一句文本描述。
7. 系统对文本描述执行情绪分析，并将结果与自评信息合并存储。
8. 系统基于健康档案、食物条目、营养数据与情绪状态进行评估。
9. 规则引擎输出问题标签、风险等级和证据说明。
10. AI 模块根据结构化结果生成更自然的建议表达。
11. 系统向用户展示评估结果、建议内容与每日总结。
12. 系统汇总历史数据，并在趋势分析页展示可视化结果。

### 4.5 核心用例设计

#### UC-01 用户注册与登录

**参与者**：普通用户
**前置条件**：用户未登录系统
**后置条件**：用户成功获得系统访问权限

**基本流程**：
1. 用户进入注册或登录页面。
2. 用户输入账号信息。
3. 系统校验信息合法性。
4. 校验通过后，系统完成注册或登录。
5. 登录成功后，系统返回认证信息并进入首页。

**异常流程**：
- 用户名已存在
- 密码错误
- 参数格式不合法
- 认证令牌失效

#### UC-02 维护健康档案

**参与者**：普通用户
**前置条件**：用户已登录
**后置条件**：用户健康档案保存成功，可用于后续评估

**基本流程**：
1. 用户进入个人档案页面。
2. 用户填写基础身体信息、运动水平和饮食目标。
3. 用户填写过敏、忌口和基础疾病提示信息。
4. 用户提交档案。
5. 系统保存档案并计算必要的派生指标。

**异常流程**：
- 身高体重为空
- 数据格式非法
- 目标值超出合理范围

#### UC-03 记录饮食并上传图片

**参与者**：普通用户
**前置条件**：用户已登录，健康档案已建立
**后置条件**：系统成功保存饮食记录并发起识别任务

**基本流程**：
1. 用户进入饮食记录页面。
2. 用户选择餐次、进食时间并填写描述。
3. 用户上传一张或多张食物图片。
4. 系统保存饮食记录。
5. 系统创建图片识别任务并返回任务状态。

**异常流程**：
- 上传图片格式不合法
- 图片大小超限
- 网络异常导致上传失败

#### UC-04 确认或修正识别结果

**参与者**：普通用户
**前置条件**：饮食记录已创建，识别任务已返回结果
**后置条件**：饮食食物项最终确认完成

**基本流程**：
1. 系统展示识别得到的食物候选结果及置信度。
2. 用户选择接受系统识别结果，或手动修改食物名称与份量。
3. 若识别结果不完整，用户可新增手动补录项。
4. 用户确认最终食物列表。
5. 系统保存确认后的食物项数据。

**异常流程**：
- 识别失败
- 返回结果为空
- 用户完全手动补录

#### UC-05 录入情绪并生成营养评估与建议

**参与者**：普通用户
**前置条件**：饮食记录已完成，食物项已确认
**后置条件**：系统成功生成营养评估、问题标签和建议内容

**基本流程**：
1. 用户进行情绪自评。
2. 用户可输入一句文本补充当前状态。
3. 系统执行文本情绪分析。
4. 系统根据饮食记录、健康档案和情绪状态完成营养评估。
5. 规则引擎生成风险等级、问题标签和证据说明。
6. AI 生成自然语言建议。
7. 系统展示评估结果、建议内容和每日总结。

**异常流程**：
- 文本情绪分析接口超时
- AI 建议生成失败
- 系统降级为仅显示规则版建议

#### UC-06 查看历史趋势与分析结果

**参与者**：普通用户
**前置条件**：用户已有历史饮食和情绪数据
**后置条件**：用户成功查看趋势分析内容

**基本流程**：
1. 用户进入趋势分析页面。
2. 系统加载历史汇总数据。
3. 系统展示热量趋势图。
4. 系统展示情绪趋势图。
5. 系统展示营养结构雷达图。
6. 系统展示情绪与高热量/高糖摄入关联图。
7. 用户查看分析结果并理解变化趋势。

**异常流程**：
- 数据不足时显示“暂无足够数据”
- 部分图表因数据缺失无法绘制

#### UC-07 管理员查看系统状态与日志

**参与者**：管理员
**前置条件**：管理员已登录
**后置条件**：管理员成功获取系统运行信息

**基本流程**：
1. 管理员进入轻量管理页面。
2. 系统展示关键接口调用次数、识别成功率和建议生成次数。
3. 管理员查看异常日志与 AI 调用日志。
4. 管理员根据 traceId 定位具体请求问题。
5. 管理员查看系统服务状态。

**异常流程**：
- 日志查询失败
- 部分统计数据延迟更新

### 4.6 异常与边界用例

#### E-01 识别结果为空

- 系统提示用户当前图片未识别出明确食物项。
- 引导用户手动补录食物名称和份量。
- 主流程不中断。

#### E-02 情绪分析接口超时

- 系统保留用户自评情绪。
- 文本情绪分析结果记为失败。
- 后续评估仍可继续执行。

#### E-03 AI 建议生成失败

- 系统直接返回规则引擎生成的基础建议文本。
- 页面提示“当前为基础建议模式”。

#### E-04 用户未完善健康档案

- 系统允许继续记录饮食。
- 但在生成评估时提示“建议完善档案以提高准确性”。

#### E-05 历史数据不足

- 趋势页正常加载。
- 数据不足的图表显示占位提示，不报错。

## 5. 数据库设计

### 5.1 数据库设计目标

本系统数据库设计围绕“采集—识别—分析—建议—追踪”闭环展开，既要支持日常业务数据的稳定存储，又要满足后续趋势分析、建议解释、日志追踪和答辩展示的需要。

### 5.2 数据库设计原则

#### 5.2.1 规范化与适度冗余结合

系统核心业务表遵循规范化设计，避免明显冗余；但在趋势统计、历史追踪和建议留档等场景下，允许保留必要快照字段，以降低查询复杂度并保证历史结果可复现。

#### 5.2.2 当前闭环优先

数据库设计优先支持毕业设计核心演示链路，不为了未来可能的复杂需求过度设计。本阶段不建立完整食物字典主表，食物条目以用户确认后的 `meal_item` 为主。

#### 5.2.3 历史可追踪

健康目标、评估结果、建议结果、识别结果和日志信息均保留必要历史记录，以满足趋势分析、问题回溯和论文展示需要。

#### 5.2.4 软删除与审计分离

核心业务表采用软删除，防止误删影响统计与历史追踪；日志表与事件表以保留原始记录为主，不进行软删除。

#### 5.2.5 前后端闭环友好

表结构应便于直接支撑前端页面和接口设计。

## 6. 接口设计

### 6.1 接口设计目标

本系统接口设计需满足以下目标：

1. 支撑主业务闭环。
2. 便于前后端分离开发。
3. 支持结果追踪与问题定位。
4. 支持异常降级。

### 6.2 接口统一约定

#### 6.2.1 接口前缀

```text
/api/v1
```

#### 6.2.2 请求与响应格式

除文件上传外，统一使用 JSON；文件上传使用 `multipart/form-data`。

#### 6.2.3 认证方式

除注册、登录等公开接口外，其余接口统一要求携带：

```http
Authorization: Bearer <token>
```

#### 6.2.4 统一响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "traceId": "20260308-abc123xyz"
}
```

#### 6.2.5 分页参数约定

请求：

```json
{
  "pageNum": 1,
  "pageSize": 10
}
```

响应：

```json
{
  "list": [],
  "total": 0,
  "pageNum": 1,
  "pageSize": 10
}
```

#### 6.2.6 时间字段约定

所有时间统一使用 ISO 8601 字符串，例如：

```text
2026-03-08T20:30:00+08:00
```

### 6.3 通用错误码设计

#### 6.3.1 认证类

- 1001：未登录或 Token 无效
- 1002：权限不足
- 1003：用户名或密码错误

#### 6.3.2 参数类

- 2001：请求参数不合法
- 2002：必填字段缺失
- 2003：数据超出合理范围

#### 6.3.3 业务类

- 3001：用户档案不存在
- 3002：饮食记录不存在
- 3003：识别任务不存在
- 3004：识别结果为空
- 3005：评估结果生成失败
- 3006：建议生成失败

#### 6.3.4 外部服务类

- 4001：视觉识别服务调用失败
- 4002：情绪分析服务调用失败
- 4003：AI 建议服务调用失败

5. 日志中不记录明文密码，AI 请求摘要做脱敏处理。

#### 7.3.3 后端分层设计

每个业务模块内部建议采用如下层次：

```text
controller
application
domain
infrastructure
dto
```

**各层职责**：
- controller：接收 HTTP 请求、参数校验、返回统一响应
- application：编排业务流程、组织事务边界

- infrastructure：数据库访问、第三方接口调用、缓存、文件存储实现
- dto：请求对象、响应对象、模块内传输对象

## 10. 项目亮点与风险控制

### 10.1 项目亮点

1. 融合饮食图像识别与情绪状态感知。
2. 前后端分离架构清晰，闭环完整。
3. 建议生成具备可解释性，不是黑盒输出。
4. 支持日志追踪与接口审计。
5. 考虑隐私保护、合规性与非医疗边界。
6. 支持趋势可视化，体现持续健康管理能力。

### 10.2 主要风险

1. JDK 版本不匹配，必须统一到 JDK 17。
2. Redis 版本偏老，不作为核心依赖，优先升级。
3. AI 接口不稳定，必须设计降级策略。
4. 建议生成过度依赖 AI，会降低可解释性。
5. 明文敏感配置存在泄露风险，必须改为 `.env` 管理。

### 10.3 风险应对策略

1. 技术栈在项目初期冻结，避免中途大调整。
2. 视觉识别失败时允许用户手动修正和补录。
3. 文本情绪分析失败时保留自评结果。
4. AI 建议失败时退回规则基础建议。
5. 核心流程优先，增强功能后置。

## 11. 结论

本总体设计文档围绕“采集—识别—分析—建议—追踪”闭环，对系统功能需求、数据库结构、接口规范、前后端模块划分、测试与验收体系进行了系统化设计。整体方案坚持“单体后端 + 前后端分离 + AI 接口接入 + 规则引擎可解释判定”的工程实现思路，能够较好满足毕业设计对系统完整性、可用性、稳定性、可解释性和可展示性的要求。
